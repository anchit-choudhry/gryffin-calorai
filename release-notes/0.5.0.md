# v0.5.0 Release Notes

**Release Date:** May 24, 2026  
**Database Schema:** v16  
**Status:** Major Feature Release

---

## Overview

v0.5.0 marks the completion of the Reminders & Notifications system (Feature 17), bringing
intelligent
time-based and day-of-week-scheduled notifications to Gryffin Calorai. Combined with the PWA
foundation
shipped in v0.4, users can now receive persistent, browser-based reminders to log meals, water
intake,
and exercise at their preferred times. This release moves the app closer to competitive parity with
leading health-tracking applications, whose reminder systems are key drivers of daily active users.

Additionally, v0.5.0 consolidates Features 4 and 15 (Diet Profiles, Recurring Meals) which were
previously
shipped in v0.4 but receive refined test coverage and integration polish in this release.

---

## Major Features

### 🔔 Feature 17: Reminders & Notifications

Schedule time-of-day notifications for daily meal logging, water intake, and exercise tracking.

**What's New:**

- `Reminder` entity: `type` (enum: meal-breakfast, meal-lunch, meal-dinner, meal-snacks, water,
  steps,
  activity), `time` (HH:MM string), `daysOfWeek` (bitmask Mon-Sun), `enabled` (boolean)
- `ReminderId` branded type in `src/types/index.ts`; `REMINDER_LABELS` map for display names
- `useReminders` hook:
  - `msUntilNextReminder()` pure function: bitmask day-of-week scheduling logic (calculates next
    occurrence given current time + day)
  - Recursive `setTimeout`-based scheduling per enabled reminder
  - `getSafeNotif()` helper: safe access to `window.Notification` API with fallback
  - Fires browser Notification on schedule; auto-reschedules for next occurrence
- `RemindersSettings` component in `src/pages/Settings.tsx`:
  - Per-reminder-type toggle (enabled/disabled)
  - Time input field (HH:MM picker or text input)
  - Day-of-week bitmask button row (Mon, Tue, Wed, Thu, Fri, Sat, Sun)
  - Permission request banner (pending/granted/denied states)
- DB schema v16: `reminders` table with unique compound index on `[userId+type]`
- `src/state/AppState.ts`: `reminders` state array, `fetchReminders`, `saveReminder`,
  `deleteReminder`
  actions; reminders loaded in `initApp()`
- `App.tsx`: `useReminders()` hook called on mount to schedule active reminders
- Comprehensive test coverage: 60 test files, 1271 tests total; **>80% coverage on all metrics**
  - `useReminders.test.ts`: 32 tests covering `msUntilNextReminder()` logic, scheduling, permission
    handling
  - `RemindersSettings.test.tsx`: 18 tests for UI interactions, state management
  - `AppState.test.ts`: 15 new tests for reminder state + actions

**Impact:** Transforms Gryffin from a passive logging tool into an active engagement driver. Users
who receive timely reminders log more consistently, improving data quality and goal adherence.

---

## Database Schema Changes

**Schema v16** (from v15):

- New `reminders` table:
  - `id` (primary key, `ReminderId` branded type)
  - `userId` (foreign key)
  - `type` (enum string: meal-breakfast, meal-lunch, meal-dinner, meal-snacks, water, steps,
    activity)
  - `time` (HH:MM 24-hour format string)
  - `daysOfWeek` (bitmask: 0-127, where bit N represents day of week N; Mon=0, Sun=6)
  - `enabled` (boolean)
  - `createdAt` (ISO 8601 timestamp)
  - Unique compound index on `[userId, type]` (one reminder per user per type)

**No breaking changes to existing tables.** All schema v15 entities (recurringMeals, dietProfiles,
etc.)
remain intact.

## Security Hardening

**Notification API Safety:**

- `getSafeNotif()` helper checks for `window.Notification` availability before access
- Graceful fallback if Notification API unavailable (e.g., insecure context)
- Permission flow follows best practices: explicit request, show denied state, allow re-request

**Data Privacy:**

- Notification payloads do not leak user data (e.g., food names, calorie counts)
- Reminders are browser-local; no telemetry or server-side scheduling

---

## Performance Improvements

- **Lazy-loaded `RemindersSettings`** component to keep initial bundle size minimal
- **Pure function `msUntilNextReminder()`** enables efficient scheduling without re-renders
- **Single `setTimeout` per reminder** (no polling or interval spam)
- **Bitmask day-of-week scheduling** avoids redundant date arithmetic

## Migration Notes

**For Existing Users (v0.4.0 → v0.5.0):**

1. Database auto-migrates from v15 to v16
2. Reminders table created empty (no historical data)
3. Reminders appear in Settings under new "Reminders" section
4. First reminder scheduled will trigger Notification permission prompt
5. Existing food logs, recipes, and progress data unaffected

**For Fresh Installs (v0.5.0 onward):**

1. Onboarding modal appears on first launch
2. Settings → Reminders tab available immediately (no reminder presets; users configure)
