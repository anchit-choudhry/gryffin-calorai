# v0.9.0 Release Notes

**Release Date:** Q3 2026 (planned)
**Database Schema:** v20 (frontend IndexedDB) + v5 (backend PostgreSQL via Flyway, planned)
**Status:** In Progress

---

## Overview

v0.9.0 ships the first wave of the UX uplift (WS A-F: three-voice typography, Almanac ornaments,
universal quick-add, diary-first dashboard, streamlined logging with null macros, and photo
logging UI scaffold), the second wave (WS G-M: retention rituals, command palette, logging speed,
system polish, onboarding, progress/nutrition viz, and PWA/haptics), and completes the AI photo
logging backend (B5). The Dexie v20 schema (photos table + FoodPhotoId) and the 9th Zustand
slice (uiSlice) ship with WS A-F.

## UX Uplift - First Wave WS A-F (Complete - June 7, 2026)

Six workstreams from `docs/canvas-design-philosophy.md`, implementing the
Phenological Ledger / Almanac design system across all surfaces.

**Test delta:** 90 test files / 1796 tests (v0.8.0) - 100 test files / 1900 tests (v0.9.0 WS A-F)
**Coverage:** statements 90.88%, functions 90.25%, branches 81.17%, lines 92.63%

### WS A - Three-voice typography (Complete)

- `Spectral` variable font added via `@font-face` in `style.css`; `--font-serif` token wired
- `SERIF_TITLE_CLS` constant exported from `lib/utils.ts`: `text-[1.35rem] font-serif font-light
  tracking-tight leading-snug text-[color:var(--ink)]`
- `SectionHeader.tsx` rewritten: serif heading + JetBrains Mono eyebrow + rule line replaces the
  previous generic bold heading; three-voice hierarchy (Spectral / Manrope / JetBrains Mono)
  activated throughout Dashboard, Recipes, Progress, and Settings
- `DateKicker.tsx` updated to JetBrains Mono micro-label style

**Architectural decision D27:** Spectral chosen over Fraunces (on the UX-principles reflex-reject
ban list) and Libre Baskerville (too neutral). Spectral's optical size range covers both `text-sm`
body use and oversized hero numerics without weight-clash against Manrope 200 thin.

### WS B - Almanac ornament activation (Complete)

- `RuleTicks.tsx` added to `components/icons/almanac/`: decorative SVG tick marks at rule
  intersections (uses `--rule` color token, `aria-hidden`)
- Paper-grain surface texture added to `style.css` as a CSS `background-image` data-URI noise
  pattern; applied via `.grain` utility class on the app shell; fallback: `background: transparent`
  for `prefers-reduced-transparency`
- All five Almanac SVG ornaments (`WheatSprig`, `MoonPhase`, `SunRay`, `RuleCorner`,
  `SeasonalFlourish`) wired into actual Dashboard and section divider positions (previously
  defined but unused)

### WS C - Universal quick-add (Complete)

- `QuickAddModal.tsx` built on native `<dialog>` element (not Radix Dialog): top-layer positioning,
  native light-dismiss via `::backdrop`, no portal hacks, native `aria-modal` semantics
- `uiSlice.ts` added as the 9th Zustand slice: `quickAddOpen: boolean`, `openQuickAdd()`,
  `closeQuickAdd()` actions; composed into `AppState.ts`
- `Cmd/Ctrl+K` global keyboard shortcut opens the modal via `useKeyboardShortcuts`
- `QuickAddFab.tsx`: center `+` FAB rendered in mobile bottom nav (`@media (pointer: coarse)`);
  `env(safe-area-inset-bottom)` padding for iPhone notch safety
- Removes all `document.querySelector` focus-hacks previously used to trigger log forms

**Architectural decision D28:** Native `<dialog>` chosen over Radix `Dialog`. The Radix primitive
adds ~4 kB to the vendor-ui chunk; native `<dialog>` gives the same top-layer guarantee (Chrome
37+, Firefox 98+, Safari 15.4+) with zero bundle cost and better `showModal()` stacking context.
All browsers in our support matrix support it.

### WS D - Diary-first dashboard (Complete)

- `Dashboard.tsx` restructured: calorie ring + macro stats remain above the fold; food log moves to
  primary position below; tracker cards (water, steps, activity, fasting) moved behind "Week" tab
- `DailyVitalsStrip.tsx` added to `components/dashboard/`: compact single-row display of the four
  tracker values with inline quick-log; expands to full tracker card on tap
- Today / Week tab strip added using roving `tabindex` and `aria-selected`; hash navigation not
  changed (still `#/`)
- All empty diary states now have inline CTA buttons ("Log your first meal") rather than passive
  placeholder text
- `EmptyState.tsx` `quiet` variant wired into the diary empty state

### WS E - Streamlined logging (Complete)

- `FoodSearchCombobox.tsx` added: Popover API + CSS anchor positioning (`lib/anchor.ts`);
  `fuzzyMatchFoodName()` from `types/index.ts` powers the suggestion list; `getRecentFoodItems()`
  IndexedDB query added to `dbService.ts` (uses `[userId+dateLogged]` compound index, most-recent
  5 entries); keyboard navigation via `aria-activedescendant`
- `useRecentFoods.ts` hook: wraps `getRecentFoodItems()` and exposes recent food results to
  `FoodSearchCombobox` as reactive state
- `FoodLogger.tsx` refactored for progressive disclosure: name + calories + meal type visible
  immediately; protein / carbs / fat / serving size behind "Add macros" expand toggle
- **Null macros:** `protein`, `carbs`, `fat` on `FoodItem` changed from `number` (required) to
  `number | undefined`; Zod schema updated (`z.number().optional()`); all render sites updated to
  show `"---"` when the value is `undefined` rather than `"0"`
- `MacroStat.tsx` updated: renders `"---"` with `text-[color:var(--ink-faint)]` when macro is
  unset; never shows `0` for genuinely unknown macros

**Architectural decision D29:** Null macros (`number | undefined`) chosen over keeping `0`
defaults. The `0` default silently corrupted macro totals (a food logged as 0g protein is
different from one with unknown protein). The `undefined` representation is propagated through
IndexedDB (field simply absent), Zustand state, and all render sites. The Zod schema uses
`.optional()` with no `.default(0)` fallback.

### WS F - Photo logging scaffold (Complete)

- `PhotoFoodLogger.tsx`: camera-capture UI with `<input type="file" capture="environment">`;
  generates a 200px-width JPEG thumbnail via `OffscreenCanvas`; stores base64 thumbnail to Dexie
  `photos` table; attaches `photoId` to the food log entry
- `ProvenanceBadge.tsx`: small inline chip showing how a food entry was logged (manual, voice,
  photo, barcode, recurring, template); uses `lucide-react` icons; `aria-label` on icon-only
  display
- `PhotoStrip.tsx`: horizontal scroll strip of meal photo thumbnails on the diary; lazy-loads
  from Dexie `photos` table; taps open a lightbox dialog
- **Dexie v20:** `photos` table added (`++id, userId, createdAt`); `FoodPhotoId` branded type
  added to `types/index.ts`; `photoId?: FoodPhotoId` added to `FoodItem` interface; migration
  from v19 is additive (no data loss)
- `FoodItem.photoId?: FoodPhotoId` wired into `FoodLogger` and `dbService.ts` CRUD methods

**Architectural decision D30:** Local Dexie v20 photos table chosen over S3 upload for WS F.
The B5 backend (S3 + vision LLM) was not yet available; a local-first photo store unblocks the UI
scaffold and photo history UX immediately. Base64 thumbnails at 200px width average ~8 kB per
photo - acceptable for typical meal-log counts. Migration to remote URLs is a field-type change
in v21+.

## B5: AI Photo Logging (Planned)

### Backend - Planned

**Photo upload + storage**

- Photo upload endpoint: `POST /api/v1/photos` - accepts multipart/form-data; stores to S3 (or
  local filesystem fallback for dev)
- Flyway migration `V5__photos.sql`: `photos` table with `id`, `user_id`, `s3_key`,
  `thumbnail_key`, `created_at`, `deleted_at`
- `PhotoService`: upload, presigned URL generation, soft-delete
- `PhotoController`: `POST /photos`, `GET /photos/{id}/url`, `DELETE /photos/{id}`

**Vision LLM proxy**

- `POST /api/v1/photos/{id}/analyze` - sends image to vision LLM (Claude claude-sonnet-4-6 or
  GPT-4V); returns structured nutrition estimate
- `AnalysisResult` DTO: `{ items: [{ name, calories, protein?, carbs?, fat?, servingSize,
  confidence }] }`
- Prompt engineering: few-shot examples for common meal types; confidence threshold gating
- `VisionAnalysisService`: wraps Anthropic/OpenAI SDK; timeout 30s; retries 2x on transient
  errors

**New env vars:**

- `S3_BUCKET`, `S3_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` (or
  `USE_LOCAL_STORAGE=true`)
- `ANTHROPIC_API_KEY` or `OPENAI_API_KEY` (one required for analysis)
- `VISION_MODEL` (default: `claude-sonnet-4-6`)

### Frontend - Planned (AI wiring)

**Wire `PhotoFoodLogger.tsx` to real backend**

- Replace local Dexie thumbnail with real `POST /api/v1/photos` upload
- Show analysis results from `POST /photos/{id}/analyze` in editable confirmation UI
- Allow per-item calorie/macro adjustment before committing to food log
- Spinner + timeout handling (analysis can take 5-15s)

## UX Uplift - Second Wave WS G-M (Planned)

Seven workstreams reusing primitives built in WS A-F.

### WS G - Retention rituals (Planned)

- Weekly Harvest review modal: summary of the week with streak, macro balance, top foods
- Milestone celebrations: confetti / `HarvestStamp` variant for achievements at 7/30/90/365-day
  streaks
- Insight cards: personalized nudges on the dashboard ("You logged dinner 6 of the last 7 days")
- Signature staggered page-load choreography: `motionVariants` expansion for page entry

### WS H - Power-user navigation (Planned)

- Command palette: `Cmd/Ctrl+K` expands the `QuickAddModal` into a full command palette with
  fuzzy search across food log, recipes, settings, and nav actions
- Past-date navigation: date picker in diary header; URL hash reflects selected date
- Global search: food item, recipe, and log-entry search across all IndexedDB tables
- Shortcut discoverability: `KeyboardShortcutsOverlay` extended with new shortcuts from WS C/H

### WS I - Logging speed (Planned)

- Copy yesterday: one-tap copy all of yesterday's log into today (with confirmation)
- Tap-to-add tracker tiles: water / step / activity quick-log directly from tracker cards
- Undo-toasts (soft-delete): 5s undo window on food log deletion; Dexie soft-delete + restore
- Portion multipliers: `x0.5`, `x1`, `x1.5`, `x2` quick-selector in `FoodLogger`
- Favorite meal combinations: `ComboMeal` entity reusing `RecurringMeals` / `MealTemplates`

### WS J - System polish (Planned)

- Skeleton extension: `extend ui/skeleton.tsx` with new skeleton variants for command palette
  and harvest review
- Sync visibility: `SyncStatusChip` expanded to show per-entity sync status
- Density toggle: compact / comfortable / spacious density tokens in `style.css`
- Microcopy voice: review and tighten all placeholder text, toast messages, empty-state copy

### WS K - Onboarding / education (Planned)

- Privacy-forward first run: explicit data-stays-local notice; Google sign-in presented as
  optional backup, not required
- Guided TDEE calibration: step-by-step wizard for age, sex, weight, activity level, goal
- Learn-as-you-go coachmarks: `ProductTourOverlay` extended for feature discovery on first use
  of new WS G-M features

### WS L - Progress / nutrition viz (Planned)

- Editorial charts: trend lines + 7-day rolling average overlay on calorie / weight charts
- Forecast line: projected weight at current calorie deficit/surplus rate (30-day horizon)
- Body visuals: silhouette with measurement annotations (waist, hip, chest) on body page
- Micronutrient heatmap: 7-day calendar view of RDA % per micronutrient

### WS M - Platform / personalization (Planned)

- PWA: `vite-plugin-pwa` + Workbox configured for offline-first caching; installable on iOS
  Safari and Android Chrome
- Haptics: `navigator.vibrate()` on log success and achievement unlock (coarse-pointer only)
- Accent theming: user-selectable accent hue (persimmon / sage / slate) via CSS custom property
- Settings IA: reorganize Settings page into tabbed layout (Account, Appearance, Goals,
  Notifications, Data)
