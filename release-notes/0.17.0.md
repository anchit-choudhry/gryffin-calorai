# v0.17.0 Release Notes

**Release Date:** June 20, 2026
**Database Schema:** v20 (frontend IndexedDB) + V22 (backend PostgreSQL via Flyway)
**Status:** Major Release

---

## Overview

v0.17.0 ships the Food DB + Barcode Lookup feature, closing Phase 1 of the competitive
strategy (see ROADMAP.md). 4.5M Open Food Facts products are imported into PostgreSQL via
a DuckDB ETL pipeline. Barcode scans now resolve nutrition data; the food search combobox
falls back to OFF full-text search when local results are empty. No frontend IndexedDB schema
changes. Backend adds Flyway V22 (two new PostgreSQL tables: `off_products` and
`off_import_log`).

## Open Food Facts Import Pipeline

### Backend - Database Migration (Complete, June 20, 2026)

**`V22__off_food_database.sql`**

- `off_products` table: `code TEXT PRIMARY KEY` (no length cap; some OFF codes exceed 20
  characters), 35 `DOUBLE PRECISION` nutrient columns (matches Parquet `DOUBLE` type; avoids
  `NUMERIC` cast overhead), stored `search_vec TSVECTOR` column updated by a `BEFORE INSERT
  OR UPDATE` trigger (sidestepping the `unaccent()` STABLE/IMMUTABLE restriction that blocks
  functional expression indexes), GIN index on `search_vec`, per-column CHECK constraints
  capping each nutrient to its physically valid range
- `off_import_log` audit table: `id`, `started_at`, `finished_at`, `rows_staged`,
  `rows_total`, `status` (`running` / `done` / `failed`); one row per import run

### Backend - Entity / Service Layer (Complete, June 20, 2026)

**`apps/backend/.../entity/OffProduct.java`**

- JPA entity annotated `@Immutable` (`org.hibernate.annotations.Immutable`) to prevent
  accidental Hibernate dirty-writes to the 4.5M-row table
- All 35 columns mapped with matching `DOUBLE` / `TEXT` / `TIMESTAMPTZ` JPA types

**`apps/backend/.../repository/OffProductRepository.java`**

- `findByCode(String code)` - exact primary-key lookup
- `searchByTsVector(String query, int limit)` - native JPQL using `plainto_tsquery` with
  `unaccent` for accent-folded full-text search

**`apps/backend/.../service/OffProductService.java`**

- `lookupBarcode(String rawCode)` - normalizes numeric EAN-8/13 codes to 13 digits with
  zero-padding before repository lookup; returns `Optional<OffProductDto>`
- `searchProducts(String query, int limit)` - delegates to FTS repository query

**`apps/backend/.../controller/OffProductController.java`**

- 2 REST endpoints, JWT-required:
  - `GET /v1/off-products/barcode/{code}` - exact barcode lookup; code normalized in service
  - `GET /v1/off-products/search?q=...&limit=...` - FTS by name/brand; `@Min(1) @Max(50)`
    on `limit` parameter; `@Validated` on controller enables Bean Validation for path/query
    params
- `ConstraintViolationException` handler added to `GlobalExceptionHandler` for `@Min`/`@Max`
  violations (returns 400 with field-level error message)

**`apps/backend/.../dto/OffProductDto.java`**

- 35-field record DTO with `@Schema` annotations; full Javadoc per Google Java Style
  Checkstyle requirements

### Backend - Import Scripts (Complete, June 20, 2026)

**`apps/backend/scripts/refresh-off-products.sh`**

- Full Parquet-to-PostgreSQL ETL pipeline:
  1. Prerequisite checks: DuckDB, psql, Parquet file exists, 5GB free in `/tmp`
  2. Orphaned staging table cleanup from previously killed runs
  3. Opens audit row in `off_import_log`
  4. DuckDB transform: deduplicates by `code` (latest `last_modified_t` wins), zero-pads
     numeric EAN-8/13 codes to 13 digits, extracts English product name from multilingual
     struct, extracts nutrient values from nested `nutriments` struct array, nulls out
     out-of-range values, strips NUL bytes and embedded newlines from text fields
  5. Loads CSV into UNLOGGED staging table (no WAL, ~30% faster bulk load)
  6. ON CONFLICT DO UPDATE with COALESCE upsert: new NULLs never overwrite existing data;
     `product_name` and `brands` always take the new value
  7. VACUUM ANALYZE
  8. Closes audit row as `done`; `trap cleanup ERR` marks `failed` on any error

**`apps/backend/scripts/cron-refresh-off.sh`**

- Monthly cron wrapper: sources `apps/backend/.env` for DB credentials; delegates to
  `refresh-off-products.sh`; writes timestamped output to rolling monthly log file
  (`OFF_LOG_DIR` override supported)
- Idempotent: running twice on the same file is safe

**`apps/backend/Dockerfile.off-importer`**

- One-shot container for running the import in Docker environments; includes DuckDB CLI,
  `psql`, and the import scripts

### Backend - Operational Runbook (Complete, June 20, 2026)

**`apps/backend/OFF-IMPORT.md`**

## Upgrade Notes

Frontend: no schema migration. `offProductApi.ts` requires a running backend with Flyway
V22 applied and OFF data imported. The barcode scan flow degrades gracefully if the backend
is unreachable (toast error, manual entry still available). The FoodSearchCombobox OFF
fallback similarly degrades to local-only results if the backend is offline.

Backend: run Flyway migration `V22__off_food_database.sql`. Docker Compose applies this
automatically on container start. The two new tables (`off_products`, `off_import_log`) are
additive; no existing tables are altered. After migration, run the OFF import pipeline per
`apps/backend/OFF-IMPORT.md` (first-time setup takes 5-15 minutes for 4.5M rows).

OFF import is not required for the app to function - it only enables barcode lookup and
the combobox fallback search. The rest of the app is unaffected if the table is empty.
