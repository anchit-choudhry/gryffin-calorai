# v0.16.0 Release Notes

**Release Date:** June 20, 2026
**Database Schema:** v20 (frontend IndexedDB) + V21 (backend PostgreSQL via Flyway)
**Status:** Minor Release

---

## Overview

v0.16.0 ships two features that together complete the privacy and Almanac artifact tracks.
The water tracker gains a 7-day tide table below the intra-day register, rendering intake
history as a proportional bar chart in the field-journal idiom. Cloud sync gains
end-to-end encryption: payloads are encrypted client-side with WebCrypto (PBKDF2 +
AES-GCM-256) before upload; the backend stores opaque blobs and cannot read user data. No
frontend DB schema changes. Backend adds Flyway V21 (two new PostgreSQL tables).

## End-to-End Encrypted Cloud Sync

### Backend - Complete (June 20, 2026)

**`V21__add_e2e_encrypted_blobs.sql`**

- `user_e2e_config` table: `user_id` (UUID PK, FK to `app_users`), `salt` (VARCHAR 255),
  `created_at`; stores the PBKDF2 salt per user so the same passphrase always derives the
  same key across devices
- `encrypted_blobs` table: `id` (UUID PK), `user_id` (FK), `client_blob_id` (VARCHAR),
  `iv` (VARCHAR 255), `ciphertext` (TEXT), `updated_at` (TIMESTAMPTZ), `is_deleted`
  (BOOLEAN); `UNIQUE (user_id, client_blob_id)` enforces client-owned IDs
- `idx_enc_blobs_user_updated` index on `(user_id, updated_at)` supports `?since=` delta
  sync queries

**`apps/backend/.../entity/EncryptedBlob.java`**

- JPA entity mapping the `encrypted_blobs` table; `@Column(columnDefinition = "TEXT")` on
  `ciphertext` to handle arbitrary-length AES-GCM output

**`apps/backend/.../dto/EncryptedBlobDto.java`**

- Record DTO for request/response; fields: `clientBlobId`, `iv`, `ciphertext`,
  `updatedAt`, `isDeleted`

**`apps/backend/.../repository/EncryptedBlobRepository.java`**

- `findByUserIdAndUpdatedAtAfter` for delta-sync pull; `deleteByUserId` for account wipe

**`apps/backend/.../service/EncryptedBlobService.java`**

- `upsertBatch`, `getDeltaBlobs`, `deleteAllForUser`, `saveSalt`, `getSalt`

**`apps/backend/.../controller/EncryptedBlobController.java`**

- 6 REST endpoints under `/v1/sync/`:
  - `GET  /e2e-config` - fetch stored salt for authenticated user
  - `POST /e2e-config` - store salt on first E2E activation
  - `POST /blobs/batch` - upsert a batch of encrypted blobs
  - `GET  /blobs` - delta pull with optional `?since=` ISO timestamp
  - `DELETE /blobs` - wipe all encrypted blobs for user
  - `POST /reset` - full account wipe (salt + blobs) for passphrase change

### Frontend - Complete (June 20, 2026)

**`apps/web/src/lib/e2eEncryption.ts`** (new file)

- Pure async WebCrypto functions; no state dependencies
- `deriveKey(passphrase, salt)` - PBKDF2 with 600,000 iterations, SHA-256, outputs
  AES-GCM-256 `CryptoKey`
- `encryptData(key, plaintext)` - AES-GCM-256 with random 12-byte IV; returns
  `{iv, ciphertext}` as Base64 strings
- `decryptData(key, iv, ciphertext)` - inverse of `encryptData`
- `exportSalt(salt)` - converts `Uint8Array` to Base64 for API transport

**`apps/web/src/lib/e2eKeyStore.ts`** (new file)

- In-memory `CryptoKey` singleton; never written to IndexedDB or localStorage
- `setKey(key)`, `getKey()`, `clearKey()` - the key is lost on page reload, requiring
  passphrase re-entry each session (by design)

**`apps/web/src/state/slices/syncSlice.ts`**

- `e2eEnabled: boolean` - persisted to `localStorage` as `gc_e2e_enabled`; survives
  reload; indicates user has activated E2E
- `e2eKeyReady: boolean` - in-memory only; never persisted; `false` after every page
  load; set `true` once passphrase is entered and key derived
- `setE2EEnabled`, `setE2EKeyReady` actions

**`apps/web/src/hooks/useSyncService.ts`**

- `unlockE2E(passphrase)` - fetches stored salt, runs `deriveKey`, sets key in
  `e2eKeyStore`, sets `e2eKeyReady: true`
- `activateE2E(passphrase)` - first-time activation: generates salt, derives key, stores
  salt via API, wipes server blobs, re-uploads all local data encrypted
- `deactivateE2E()` - clears key store, sets `e2eEnabled: false`, wipes server blobs
- Push path: if `e2eKeyReady`, serializes payload to JSON, encrypts via
  `e2eEncryption.ts`, sends `EncryptedBlobDto`; otherwise uses existing plaintext sync
- Pull path: if `e2eKeyReady`, decrypts each blob after download before writing to
  IndexedDB

**`apps/web/src/components/CloudSyncPanel.tsx`**

- 4-state passphrase state machine:
  - `not-connected` - not logged in; shows login prompt
  - `connected-no-e2e` - logged in, E2E not enabled; shows enable button
  - `connected-e2e-locked` - E2E enabled but key not in memory; shows unlock passphrase
    form
  - `connected-e2e-ready` - key loaded; shows sync status and disable button
- Passphrase entry includes a confirmation field on first activation

**`apps/web/src/types/index.ts`**

- `SyncBlobPayload` interface: `clientBlobId`, `iv`, `ciphertext`, `updatedAt`,
  `isDeleted`

## Upgrade Notes

Frontend: no schema migration. `e2eEnabled` and `e2eKeyReady` are new `syncSlice` fields
with safe defaults (`false`); existing users are unaffected.

Backend: run Flyway migration `V21__add_e2e_encrypted_blobs.sql`. Docker Compose applies
this automatically on container start. The two new tables (`user_e2e_config`,
`encrypted_blobs`) are additive; no existing tables are altered. E2E sync endpoints
require the user to be authenticated (JWT); unauthenticated requests return 401.
