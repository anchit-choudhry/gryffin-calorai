# v0.11.0 Release Notes

**Release Date:** June 14, 2026
**Database Schema:** v20 (frontend IndexedDB) + v4 (backend PostgreSQL via Flyway)
**Status:** Major Release

---

## Overview

v0.11.0 adds an adaptive intelligence layer that learns from the user's own food and weight data
with no cloud API cost: an observed-TDEE engine with EMA smoothing, plateau detection, Pearson-r
correlation insights, and a week-end energy-balance forecast. Data portability ships as a
multi-source CSV importer (MyFitnessPal, Cronometer, Lose It) and an Apple Health XML importer,
demolishing the switching cost from every major competitor. Meal pattern suggestions complete the
friction-killer cluster, proactively surfacing "Usually: oats - tap to log" chips based on
frequency analysis of the user's log history. A security hardening pass fixed a mixed-radix SSRF
bypass in the recipe importer and added date and calorie validation guards in all CSV parsers.

## Adaptive Intelligence

### Frontend - Complete (June 14, 2026)

**`apps/web/src/lib/adaptiveTdee.ts`**

- `computeAdaptiveTdee(logs, measurements)`: EMA-smoothed observed TDEE derived from the user's
  own food logs and weight history; no external API required; surfaces "your measured TDEE"
  alongside the Mifflin-St Jeor formula estimate
- `detectPlateau(measurements, goalType)`: flags a plateau when the 21-day weight change is under
  0.5 kg while a deficit goal is active; returns `{ isPlateau: boolean; daysSinceChange: number }`
- `computeWeeklyForecast(logs: readonly FoodLogEntry[])`: projects end-of-week calorie surplus or
  deficit from week-to-date entries; parameter typed as the structural `FoodLogEntry` shape
  (`{ calories: number; dateLogged: string }`) rather than the full `FoodItem` to keep the library
  decoupled from DB types

**`apps/web/src/lib/correlations.ts`**

- `computeCorrelations(state)`: Pearson-r computation for three data pairs:
  - Sodium vs next-day weight (water retention signal)
  - Training days vs calorie adherence
  - Fasting duration vs total daily intake
- Returns `CorrelationResult[]` with `label`, `r` coefficient, and a
  `"positive" | "negative" | "none"` direction tag; only shown when 14+ days of paired data
  exists to avoid spurious correlations on sparse history

**`apps/web/src/components/progress/AdaptiveTdeePanel.tsx`**

- Displays observed TDEE alongside Mifflin-St Jeor estimate with a "Measured vs formula" delta
- Opt-in toggle in `settingsSlice` to use the observed value as the active calorie goal driver
- Plateau alert: `InsightCard` rendered when `detectPlateau` returns `{ isPlateau: true }`

**`apps/web/src/components/progress/CorrelationInsightsPanel.tsx`**

- Lists `CorrelationResult` items color-coded by direction; sparse-data guard hides the panel
  until 14 days of paired observations are available

**`apps/web/src/components/progress/EnergyForecastCard.tsx`**

- Week-end surplus/deficit projection card in the Progress page; uses `computeWeeklyForecast`;
  shows projected kcal balance and estimated weight impact in plain language

**`apps/web/src/components/progress/ProjectedWeightCard.tsx`**

- Projects body weight at end of week based on current weekly energy balance using the 3,500
  kcal-per-pound heuristic; displayed alongside the actual weight trend sparkline

## Data Portability - CSV Import

### Frontend - Complete (June 14, 2026)

**`apps/web/src/lib/importers/utils.ts`**

- `isISODate(s)`: type guard validating `YYYY-MM-DD` format and calendar correctness (rejects
  `2026-02-30` and similar); used by all three CSV parsers before any date is written to IndexedDB
- `parseNum(s)`: trims and parses numeric strings; returns `undefined` on NaN; guards against
  `Infinity` from `parseFloat("Infinity")` via `Number.isFinite()` check
- `splitCsvRow(line)`: RFC 4180 CSV tokenizer handling quoted fields, escaped inner quotes, and
  commas within quoted strings
- `normalizeMealType(s)`: maps common source names to the app's `MealType` union; returns
  `undefined` for unrecognized values rather than silently defaulting

**`apps/web/src/lib/importers/mfp.ts`**

- `parseMfpCsv(csv)`: parses MyFitnessPal food diary CSV export; maps Date, Meal, Food Name,
  Calories, Protein (g), Carbohydrates (g), Fat (g) columns; skips MFP summary rows; returns
  `ParseResult { entries: ImportedFoodEntry[]; skippedRows: number }`

**`apps/web/src/lib/importers/cronometer.ts`**

- `parseCronometerCsv(csv)`: parses Cronometer export; Day, Food Name, Energy (kcal), Protein (g),
  Carbs (g), Fat (g), Sodium (mg), Fiber (g) columns

**`apps/web/src/lib/importers/loseit.ts`**

- `parseLoseItCsv(csv)`: parses Lose It! diary CSV; Date, Name, Type, Calories, Protein (g),
  Carbohydrates (g), Fat (g), Sodium (mg), Fiber (g) columns; Type column wired to
  `normalizeMealType` for meal type preservation

**`apps/web/src/components/settings/CsvImportPanel.tsx`**

- Format selector (MyFitnessPal / Cronometer / Lose It); file input; import summary with entry
  count and skipped-row count
- Each `dateLogged` validated by `isISODate()` before dispatch; invalid dates are counted as
  skipped rows, not silent failures
- Success toast via sonner with entry and skip counts

## Data Portability - Apple Health XML Import

### Frontend - Complete (June 14, 2026)

**`apps/web/src/lib/importers/appleHealth.ts`**

- `parseAppleHealthXml(xml)`: parses Apple Health `export.xml`; extracts
  `HKQuantityTypeIdentifierBodyMass` (weight in kg) and `HKQuantityTypeIdentifierStepCount`
  records; returns `{ bodyMeasurements: ImportedBodyMeasurement[]; stepLogs: ImportedStepLog[] }`
- Missing or malformed `<Record>` elements skipped gracefully; invalid dates filtered via
  `isISODate()` before DB write

**`apps/web/src/components/settings/AppleHealthImportPanel.tsx`**

- Drag-and-drop or file-select for the Apple Health `export.xml` file; progress indicator;
  dispatches to `addBodyMeasurement` and `addStepLog` for each parsed record
- File input value reset after import (`inputRef.current.value = ""`) so the same file can be
  re-imported if needed
- Duplicate measurements on the same date are skipped with a warning count in the result summary

## Meal Pattern Suggestions

### Frontend - Complete (June 14, 2026)

**`apps/web/src/lib/mealPatterns.ts`**

- `analyzeMealPatterns(logs, selectedDate)`: frequency analysis over the past 14 days; surfaces
  foods logged for the same `MealType` 3+ times as suggestions; returns
  `MealPatternSuggestion[] = { name, mealType, frequency, lastLogged }` sorted by frequency
- Time-of-day relevance: Breakfast suggestions shown before 10:00, Lunch before 14:00, etc.

**Dashboard integration**

- Suggestion chips appear above the relevant meal section at the appropriate time of day;
  tapping pre-fills the Quick Add modal with food name and meal type
- Chips hidden when fewer than 3 days of log history exist to avoid noise for new users

## Security Hardening

### Frontend - Complete (June 14, 2026)

Applied during code review of v0.11.0 additions.

**`apps/web/src/hooks/useRecipeImport.ts` - SSRF: mixed-notation IP bypass**

- `parseOctet(s)` helper added: per-octet parser that handles hex (`0x` prefix), octal (`0`
  prefix), and decimal notations; `isBlockedHost()` now calls `parseOctet` on each IP octet,
  blocking `http://0x7f.0.0.1`, `http://0177.0.0.1`, and other mixed-radix representations
- Previously, `parseInt(s, 10)` silently parsed hex/octal octets as `0`, allowing SSRF bypass
  via mixed-radix IP notation past the private-IP blocklist

**`apps/web/src/lib/importers/utils.ts` - Date injection guard**

- `isISODate()` type guard validates format and calendar correctness before any imported date is
  written to IndexedDB; prevents invalid or adversarially crafted dates from reaching the DB layer

**CSV parsers - calorie overflow guard**

- `Number.isFinite(cal)` check alongside `cal === undefined` in all three parsers (`mfp.ts`,
  `cronometer.ts`, `loseit.ts`); blocks `Infinity` (produced by `parseFloat("Infinity")`) from
  entering the calorie field; non-finite entries counted as skipped rows

  `"persimmon" | "sage" | "indigo" | "amber" | "rose"` (removes `slate`, `ocean`, `ember`);

  `inputRef.current.value = ""` in the import completion handler so the same file can trigger

- `adaptiveTdee.ts`: `computeWeeklyForecast` parameter type widened from `readonly FoodItem[]` to
  `readonly FoodLogEntry[]` (structural type `{ calories: number; dateLogged: string }`); removes
  the transitive dependency on the full `FoodItem` DB type from the analytics library

  - `@zxing/browser` and `@zxing/library`: replaced with the native `BarcodeDetector` Web API

    `vite.config.ts`

    (`Math.floor((Date.now() - new Date(startTime).getTime()) / 1000)`); `vendor-date` manual
    chunk entry removed from `vite.config.ts`
  - `autoprefixer`: PostCSS plugin removed from `postcss.config.js`; Tailwind v4 handles vendor
    prefixing internally via its Vite plugin
  - `@fontsource-variable/public-sans`: dead import with no usages; removed declaration from
    `fontsource.d.ts`

## Upgrade Notes

No breaking changes or schema migrations required. All new features are additive:

- Adaptive TDEE opt-in toggle available in Settings (off by default; Mifflin-St Jeor remains
  the default calorie goal driver)
- CSV import panel added to the Settings Import section; no schema migration required
- Apple Health XML import added to the Settings Import section; no schema migration required
- Meal pattern suggestions appear automatically on the Dashboard after 3 days of log history

**`uiSlice` localStorage migration:** Existing `gc_accent` values of `"slate"`, `"ocean"`, or
`"ember"` are automatically migrated to their replacements on first load: `"slate"` - `"indigo"`,
`"ocean"` - `"indigo"`, `"ember"` - `"amber"`. The migrated value is written back to
`localStorage` immediately, so the migration runs once per affected device.
