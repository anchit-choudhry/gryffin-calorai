# v0.21.0 Release Notes

**Release Date:** June 28, 2026
**Database Schema:** v21 (frontend IndexedDB, unchanged) + V6 (backend, newest Flyway migration)
**Status:** Minor Release

---

## Overview

v0.21.0 ships the two features deferred from v0.20.0: backend REST sync for recipes, meal
templates, reminders, and diet profiles (Flyway V5); and real Web Push notifications via VAPID
(Flyway V6). Both tracks are paired with SSRF hardening on the push endpoint. The release also
closes production quality gaps: the RecipeShareSheet now handles async failures gracefully,
the fflate dependency is fully removed in favor of Web Streams API and JSZip, and the Vite
preCompress plugin is extracted for maintainability.

---

## Track 1 - Backend sync REST for recipes, meal templates, reminders, and diet profiles

### Problem

v0.20.0 wired the frontend to enqueue sync operations for these four entity types, but the
backend had no REST endpoints to receive them. Any sync flush against the server silently
discarded recipe and template data.

### Backend - Complete (June 28, 2026)

**Flyway `V5__recipe_sync.sql`**

- New `recipes` table: UUID PK, `user_id` FK, `ingredients` JSONB, soft-delete `deleted_at`,
  `idx_recipes_user_updated` compound index on `(user_id, updated_at)`.
- New `meal_templates` table: UUID PK, `user_id` FK, `foods` JSONB, soft-delete,
  `idx_meal_templates_user_updated` index.
- New `user_reminders` table: UUID PK, `user_id` FK, `type`, `time`, `days_of_week` bitmask,
  `enabled` flag, soft-delete, `idx_user_reminders_user_updated` index.
- New `user_diet_profiles` table: UUID PK, `user_id` UNIQUE FK (singleton per user), `preset`,
  `restrictions` JSONB text array, soft-delete, `idx_user_diet_profiles_user` index.

**Entities, repositories, DTOs**

- `Recipe`, `RecipeIngredient`, `MealTemplate`, `MealTemplateFoodEntry`, `UserReminder`,
  `UserDietProfile` JPA entities with `@JsonIgnore` on `user` relation.
- `RecipeRepository`, `MealTemplateRepository`, `UserReminderRepository`,
  `UserDietProfileRepository` Spring Data JPA repositories with `findByUserId` queries.
- `RecipeDto`, `MealTemplateDto`, `ReminderDto`, `DietProfileDto` request/response records
  with full `@Schema` Javadoc coverage and `@NotNull`/`@Valid` validation.

**Services**

- `RecipeService` - list/upsert/soft-delete recipes per user; upsert matches on `syncId`.
- `MealTemplateService` - list/upsert/soft-delete meal templates per user.
- `ReminderService` - list/upsert/delete reminders per user.
- `DietProfileService` - upsert singleton diet profile per user (INSERT ON CONFLICT UPDATE).

**Controllers**

- `RecipeController` - `GET/POST /v1/sync/recipes`, `DELETE /v1/sync/recipes/{id}`.
- `MealTemplateController` - `GET/POST /v1/sync/meal-templates`,
  `DELETE /v1/sync/meal-templates/{id}`.
- `ReminderController` - `GET/POST /v1/sync/reminders`, `DELETE /v1/sync/reminders/{id}`.
- `DietProfileController` - `GET/PUT /v1/sync/diet-profile`.
- All controllers secured by `@PreAuthorize("isAuthenticated()")` and extract `userId` from
  the JWT principal via `SecurityConfig.userId(auth)`.

**`apps/backend/.../service/UserDataResetService.java`** (new)

- Extracted from `EncryptedBlobController` into a standalone `@Transactional` service.
- `deleteAllByUserId(UUID)` coordinates deletion across 11 entity types in a single
  transaction: encrypted blobs, food logs, water logs, activity, body, step, fasting,
  plus the four new types (recipes, meal templates, reminders, diet profile).
- `EncryptedBlobController` constructor simplified from 12 injected repositories to 2
  (blob repo + reset service).

## Track 2 - Web Push / VAPID notifications

### Problem

`useReminders` delivered notifications via `setTimeout`, which fires only while the browser tab
is open. Users who closed the tab missed every reminder.

### Backend - Complete (June 28, 2026)

**Flyway `V6__push_subscriptions.sql`**

- New `push_subscriptions` table: UUID PK, `user_id` FK, `endpoint`, `p256dh`, `auth`, optional
  `timezone`, UNIQUE constraint on `(user_id, endpoint)`,
  `idx_push_subscriptions_user_id` index.

**`PushNotificationService.java`**

- Self-implemented VAPID using `JJWT` (JWT signing) and `Bouncy Castle` (ECDH key generation
  and P-256 point math). No `nl.martijndwars:web-push` third-party library (supply-chain risk
  eliminated).
- `sendPush(sub, payload)` builds the VAPID JWT authorization header, encrypts the payload
  using ECDH + AES-128-GCM (RFC 8291), and POSTs to the subscriber's push endpoint via
  Spring `RestClient`.
- **SSRF hardening** - runtime guard: `URI.create(sub.getEndpoint()).getScheme()` must equal
  `"https"` before any outbound request is made. Logs a warning and returns early if not.

**`PushSubscriptionDto.java`**

- `@Pattern(regexp = "^https://.*", message = "Endpoint must use HTTPS")` on the `endpoint`
  field - schema-level SSRF guard that rejects `http://` and `file://` endpoints at
  deserialization time before they reach the service.

**`PushSubscriptionController.java`**

- `POST /v1/push/subscribe` - stores subscription for the authenticated user; deduplicates by
  endpoint.
- `DELETE /v1/push/unsubscribe` - removes a specific subscription.
- `GET /v1/push/vapid-public-key` - returns the VAPID public key (base64url) needed by the
  browser to subscribe.
- `VapidPublicKeyDto` response record.

### Frontend - Complete (June 28, 2026)

**`apps/web/src/lib/pushNotifications.ts`**

- `subscribePush()` - requests browser push permission, subscribes via
  `registration.pushManager.subscribe`, and POSTs the subscription to the backend.
- `unsubscribePush()` - unsubscribes at the browser level and DELETEs from the backend.
- `isPushSupported()` - feature-detects `PushManager` + service worker availability.

**`apps/web/public/sw.js`**

- `push` event listener: parses the JSON payload and calls
  `self.registration.showNotification(title, options)` to display the system notification.
- `notificationclick` listener: focuses the existing app window or opens a new one.

**`apps/web/src/hooks/useReminders.ts`**

- Integrates push subscription status alongside the existing `setTimeout` fallback.
- `useReminders` now exports `isPushEnabled` and `togglePush()` for the Settings UI.

## Bug Fixes

- `RecipeShareSheet`: silent `Promise` rejection no longer leaves the share URL empty without
  any user feedback. Added `shareError` state; the caption shows "Unable to generate share
  link." on failure.
- `RecipeShareSheet`: Copy Link and Share buttons were active while the URL was still
  generating (first ~100ms). Both buttons are now `disabled={!shareUrl}` with
  `disabled:pointer-events-none disabled:opacity-40`.
- `recipeShare.ts`: `uint8ToBase64url` switched from `String.fromCharCode(...bytes)` spread
  to a `for...of` loop. The spread form crashes with a stack overflow on large recipes
  (argument list too long for `Function.apply`).

## Technical Debt / Cleanup

- Replaced `fflate` in `apps/web/src/lib/recipeShare.ts` with the native Web Streams API
  (`CompressionStream("deflate-raw")` / `DecompressionStream`). No runtime dependency for
  deflate; both `encodeSharePayload` and `decodeSharePayload` are now `async`.
- Replaced `fflate` in `apps/web/src/hooks/useDataExport.ts` with `jszip`. ZIP export now
  uses `JSZip` + `generateAsync("blob")`.
- Extracted the preCompress Vite plugin from `apps/web/vite.config.ts` into
  `apps/web/plugins/preCompress.ts`. Config file is now readable; plugin is independently
  testable.
- Fixed dead `manualChunks` vendor entry: `class-variance-authority` (removed dep) replaced
  with `tailwind-variants` in the `vendor-ui` chunk definition in `vite.config.ts`.
- `tw-animate-css` package removed; keyframe animations inlined directly into `style.css`.
- Apple Health importer (`appleHealth.ts`): tightened XML parsing for edge-case date formats;
  improved test coverage in `appleHealth.test.ts` for missing/malformed fields.

## Upgrade Notes

**Flyway out-of-order migrations** - V5 and V6 were added after V21 and V22 already existed
in the migration table. A new clean-install database runs V1-V6, then V21-V22 correctly. An
existing deployed database that already has V21 and V22 applied will reject V5 and V6 unless
`spring.flyway.out-of-order=true` is set in `application.yml`. This is already configured in
`apps/backend/src/main/resources/application.yml`.

**VAPID key pair** - The push notification service requires a VAPID key pair at startup. Set
`VAPID_PUBLIC_KEY` and `VAPID_PRIVATE_KEY` in `apps/backend/.env`. Generate with:

```bash
npx web-push generate-vapid-keys
```

No frontend IndexedDB schema change (stays at v21). No `pnpm install` needed beyond what was
already done for v0.20.0 unless pulling fresh.
