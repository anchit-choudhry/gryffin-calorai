# v0.8.0 Release Notes

**Release Date:** June 7, 2026
**Database Schema:** v19 (frontend IndexedDB) + v4 (backend PostgreSQL via Flyway)
**Status:** Major Release

---

## Overview

v0.8.0 delivers bidirectional cloud sync, connecting the offline-first frontend to the
Spring Boot backend introduced in v0.7.0. It also ships the UI v2 Almanac Refresh (Almanac /
Field Journal aesthetic), backend security hardening, and developer tooling infrastructure. AI
Photo Logging ships in v0.9.0.

The cloud sync architecture is offline-first: every write succeeds immediately to IndexedDB, then a
background sync queue flushes changes to the server. Pulling uses `?since=<ISO timestamp>` delta
endpoints on each entity so only changed records are transferred.

## Cloud Sync (Complete)

### Backend - Complete (May 26, 2026)

**Flyway migration `V4__sync_support.sql`**

- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()` and `deleted_at TIMESTAMPTZ` added to all 6 log
  tables (`food_items`, `water_logs`, `activity_logs`, `body_measurements`, `step_logs`,
  `fasting_sessions`)
- Delta-sync indices: `CREATE INDEX idx_<entity>_user_updated ON <table>(user_id, updated_at)`
  on each table for O(log n) `?since=` queries

**Entities, repositories, services, controllers**

- All 6 entities updated: `updatedAt`, `deletedAt`, `@PreUpdate touch()`, `setId(UUID)` for
  upsert-by-client-ID semantics (Spring Data `save()` calls JPA `merge()` when id is non-null)
- All 6 repositories: `findByUserId...DeletedAtIsNull` (soft-delete filtering) +
  `findByUserIdAndUpdatedAtAfter` (delta sync)
- 5 new DTOs (`ActivityLogDto`, `BodyMeasurementDto`, `StepLogDto`, `FastingSessionDto`,
  `TdeeProfileDto`) + `FoodItemDto`/`WaterLogDto` updated with `updatedAt`/`deletedAt`
- 7 services: `FoodItemService` (upsert + soft-delete + `getChangesSince`), `WaterLogService`,
  `ActivityLogService`, `BodyMeasurementService`, `StepLogService`, `FastingSessionService`,
  `TdeeProfileService`
- 7 REST controllers: full CRUD + `GET /changes?since=<Instant>` on each entity type;
  `TdeeProfileController` is 1:1 per user (`GET /api/v1/tdee-profile`, `PUT /api/v1/tdee-profile`)
- 6 service unit test files (46 tests total)

### Frontend - Complete (May 26-27, 2026)

**`apps/web/src/lib/apiClient.ts`**

- JWT-aware HTTP client; stores `gc_access_token`, `gc_refresh_token`, `gc_expires_at` in
  `localStorage`; auto-refreshes 60s before expiry; `ApiError` class with HTTP status; typed
  `api.get/post/put/delete` helpers; `api.auth.exchangeToken(provider, idToken)` and
  `api.auth.logout(refreshToken)`; `isAuthenticated()` and `clearTokens()` exports

**Dexie schema v19**

- `syncId` index added to 6 syncable tables (`foodItems`, `waterLogs`, `activityLogs`,
  `bodyMeasurements`, `stepLogs`, `fastingSessions`)
- New `syncQueue` table + `SyncQueueEntry` type (`entityType`, `operation`, `syncId`, `payload`,
  `retries`, `createdAt`)
- `syncId?: string` and `deletedAt?: string` added to all 6 entity interfaces

**`apps/web/src/state/slices/syncSlice.ts`**

- New Zustand slice: `syncStatus` (`"idle"|"syncing"|"synced"|"offline"|"error"`), `lastSyncedAt`
  (persisted to `localStorage` as `gc_last_synced_at` for cross-session delta sync), `syncError`
- `setSyncStatus`, `setLastSyncedAt`, `setSyncError` actions

**`apps/web/src/hooks/useSyncService.ts`**

- Pulls delta changes from all 6 `/changes?since=` endpoints; merges by `syncId` (update
  existing record, add new, hard-delete tombstones)
- Pulls TDEE profile via `GET /api/v1/tdee-profile` and upserts local record
- Flushes `syncQueue` before pulling (push-then-pull ordering)
- Triggers on mount, every 5 min (`setInterval`), `online` event, `focus` event, and `gc:sync`
  custom DOM event
- Exports `enqueueSyncOperation()` for use by state slices

**Push-on-write in all 5 state slices**

Each create/update/delete operation now calls `enqueueSyncOperation()` after the Dexie write:

- `foodSlice`: add/update/delete food item, toggleFavorite
- `trackerSlice`: add/delete water log, add/delete step log
- `bodySlice`: add/update/delete body measurement
- `activitySlice`: add/delete activity log, start/end fasting session
- `settingsSlice`: save TDEE profile (sentinel `syncId: "tdeeProfile"`, `operation: "create"` for
  server-side upsert via `PUT /api/v1/tdee-profile`)

`syncId = crypto.randomUUID()` is generated client-side at record creation and stored in Dexie.
`enqueueSyncOperation` is a no-op when `isAuthenticated()` returns false.

**`apps/web/src/components/SyncStatusChip.tsx`**

- Navbar sync status indicator; hidden when not authenticated
- Shows icon + label for each `SyncStatus`: spinning loader (syncing), cloud (synced), wifi-off

**`apps/web/src/components/CloudSyncPanel.tsx`**

- Sign-out: `api.auth.logout()` - `clearTokens()` - `window.location.reload()`

## Auth Hardening (May 2026)

### Backend - Complete (May 2026)

**`AuthController.java` - logout endpoint**

- `POST /auth/logout` endpoint added; accepts `LogoutRequest` DTO containing the refresh token JTI
- Delegates to `AuthService.logout()` which deletes the JTI row from the `refresh_tokens` table;
  subsequent reuse of the same JTI is rejected with 401

**`RefreshTokenCleanupTask.java`**

- `@Scheduled` Spring task purging expired rows from `refresh_tokens` where `expires_at < NOW()`
- Prevents unbounded table growth without manual intervention
- Wired via `@EnableScheduling` on `GryffinCaloraiApplication.java`

## Known Limitations

- Cloud sync requires `VITE_GOOGLE_CLIENT_ID` to be set; app remains fully offline-first without it
- Sync is one-directional for recipes and achievements (pull not yet implemented); food logs, water,
  steps, body measurements, activity, fasting, and TDEE profile are fully bidirectional
- AI Photo Logging: frontend scaffold and backend move to v0.9.0
- Barcode lookup (Feature 12) remains deferred; scanner UI exists but no nutrition API integration
- USDA micronutrient data auto-import remains planned; manual entry only
