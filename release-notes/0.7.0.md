# v0.7.0 Release Notes

**Release Date:** May 25, 2026
**Database Schema:** v18 (frontend IndexedDB) + v3 (backend PostgreSQL via Flyway)
**Status:** Major Release

---

## Overview

v0.7.0 is the largest release to date, delivering three parallel workstreams in a single milestone:

1. **Remaining v0.6 frontend features** - Micronutrient Tracking (Feature 14), Conflict Summary UI
   (W1), WCAG accessibility improvements (W2), and Zustand store slicing (W3)
2. **Monorepo restructure** (M0) - repository reorganized into `apps/web/`, `apps/backend/`,
   `apps/ios/`, `apps/android/`, and `packages/`
3. **Backend scaffold** - Spring Boot 3.5 + PostgreSQL + OpenAPI codegen + OAuth2/JWT auth
   foundation

This release marks the transition from a pure client-side SPA to a full-stack, multi-platform
project ready for cloud sync (v0.8) and native mobile (v0.9-v1.0).

## Major Features

### Feature 14: Micronutrient Tracking (~25 items)

Track 25 essential micronutrients alongside macros, with RDA percentage gauges on the Progress page.

**What's New:**

- DB schema v18: `nutritionData` flexible field on `FoodItem` stores vitamins, minerals, and other
  nutrients; migration converts existing flat macro fields
- `src/lib/micronutrientRDA.ts`: RDA constants for 25 micronutrients (vitamins A, B6, B12, C, D, E,
  K, folate, niacin, thiamine; minerals calcium, iron, magnesium, potassium, sodium, zinc, copper,
  iodine, phosphorus, selenium; fiber, sugar, saturated fat, trans fat, cholesterol)
- `src/db/nutrition.ts`: dedicated nutrition DB helpers with compound index queries
- `useMicronutrientData` hook: computes daily totals and RDA percentages per nutrient
- `MicronutrientPanel` component on Progress page: gauge-style progress bars for key micronutrients,
  color-coded by deficiency/excess, togglable expanded view
- `FoodLogger` form: expandable "Advanced Nutrition" section with per-nutrient inputs

**Database Changes:**

| Version | Change                                                                                                       |
|---------|--------------------------------------------------------------------------------------------------------------|
| v18     | `nutritionData: Record<string, number>` field on `foodItems`; migration from `protein/carbs/fat` flat fields |

### W1: Conflict Summary UI for Data Import

When importing a backup, the app now detects and surfaces conflicts before writing any data.

**What's New:**

- `DataImportConflictModal` component: lists conflicting food items, recipes, and activity logs with
  field-level diffs
- Per-item resolution choices: skip, overwrite, or merge (batch-apply radio buttons)
- Conflict detection integrated into `useDataImport` hook pre-import phase
- Post-import summary toast: "X conflicts resolved, Y items imported"
- `conflictResolutionMode` state in the import slice

---

### W2: Accessibility Improvements

Reduced-motion support and accessibility utilities across all animated components.

**What's New:**

- `src/lib/a11y.ts`: `useReducedMotion()` hook (wraps `prefers-reduced-motion` media query),
  heading hierarchy validator, focus ring utilities
- All motion 12 animated components updated to skip or reduce animations when `useReducedMotion()`
  returns true
- Form label audit: all `FormField` components verified to have associated labels or ARIA labels
- Focus ring enhancements on interactive elements across Dashboard, Progress, Recipes, and Settings
- `src/lib/a11y.test.ts`: unit tests for all accessibility utilities

---

### W3: Zustand Store Slicing Refactor

`AppState.ts` decomposed from ~2,500 lines into 7 focused slice files.

**What's New:**

- `apps/web/src/state/slices/`:
  - `foodSlice.ts` - food items, favorites, nutrition data
  - `recipeSlice.ts` - recipes, meal templates, meal plans
  - `bodySlice.ts` - body measurements, TDEE profile
  - `activitySlice.ts` - activity logs, fasting sessions
  - `trackerSlice.ts` - water logs, step logs, streaks
  - `settingsSlice.ts` - theme, reminders, diet profile, recurring meals
  - `coreSlice.ts` - user, onboarding, achievements, loading/error states
- `AppState.ts` composes all slices; `useAppState` API unchanged (no consumer breakage)
- Each slice independently testable; cross-slice actions handled via `coreSlice`

### Spring Boot 3.5 + PostgreSQL Backend

Production-ready backend scaffold with Docker Compose, Flyway migrations, and Actuator.

**What's New:**

- `apps/backend/`: Maven project, Spring Boot 3.5.0, Java 21
- **Entities:** `AppUser`, `FoodItem`, `WaterLog`, `BodyMeasurement`, `StepLog`, `ActivityLog`,
  `FastingSession`, `TdeeProfile`
- **Controllers:** `AuthController` (login/register/refresh), `FoodItemController` (CRUD),
  `HealthController` (actuator passthrough)
- **Services + Repositories:** JPA repositories with custom queries; service layer for
  business logic
- **Database:** Flyway migration `V1__init.sql` creates all tables with proper indices and
  foreign keys (mirrors IndexedDB schema)
- **Docker Compose:** `docker compose up -d` starts PostgreSQL 5432, PgAdmin 5050, backend 8080
- **Config:** `.env.example` with all required variables (`DATABASE_URL`, `DATABASE_USER`,
  `DATABASE_PASSWORD`, `JWT_SECRET`, `GOOGLE_CLIENT_ID`, `CORS_ALLOWED_ORIGINS`)
- **Actuator:** health endpoint at `/actuator/health`

### Auth Scaffold (OAuth2 + JWT)

Spring Security foundation for Google OAuth2 and JWT-based authentication.

**What's New:**

- Spring Security OAuth2 resource server configuration
- JWT generation and validation using `jjwt` 0.12.6 (HS256, secret from `JWT_SECRET` env var)
- `AuthController`: `/auth/register`, `/auth/login`, `/auth/refresh` endpoints
- Google OAuth2 client ID wired via `GOOGLE_CLIENT_ID` env var (web, Android, iOS)
- CORS configured via `CORS_ALLOWED_ORIGINS` env var
- Token expiry: access token 1 h, refresh token 30 d

## Database Schema Changes

### Frontend (IndexedDB / Dexie.js)

| Version | Change                                                                                                                           |
|---------|----------------------------------------------------------------------------------------------------------------------------------|
| v18     | `nutritionData: Record<string, number>` added to `foodItems`; migration populates from existing `protein`, `carbs`, `fat` fields |

### Backend (PostgreSQL / Flyway)

| Migration                                 | Description                                                                                                                                                        |
|-------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `V1__init.sql`                            | Initial schema: users, food_items, water_logs, body_measurements, step_logs, activity_logs, fasting_sessions, tdee_profiles                                        |
| `V2__fix_provider_subject_constraint.sql` | Replaces single-column `provider_subject` UNIQUE with composite `(provider, provider_subject)`; adds `chk_food_items_meal_type` CHECK constraint on food_items     |
| `V3__refresh_tokens.sql`                  | Creates `refresh_tokens` table with `jti` UUID PK, `user_id` FK, `expires_at`; indices on `user_id` and `expires_at`; enables single-use JTI token rotation |

## Migration Notes

**For Existing Frontend Users (v0.6.0 -> v0.7.0):**

1. Database auto-migrates from v17 to v18 on first load
2. All existing food items receive a `nutritionData` object pre-populated with their
   existing calorie, protein, carbs, and fat values
3. No data loss; all logs, recipes, and settings unaffected

**For Backend Setup (first time):**

1. `cd apps/backend && cp .env.example .env` - fill in `JWT_SECRET` (min 256 bits)
2. `docker compose up -d` - starts PostgreSQL, PgAdmin, and backend
3. Backend auto-runs Flyway migrations on startup
4. Frontend remains fully functional without the backend (IndexedDB-first)

**For Dev Environment (monorepo):**

1. Run `pnpm install` from repo root (workspace hoisting picks up all packages)
2. All existing commands (`pnpm dev`, `pnpm test`, `pnpm build`) continue to work from root
3. Backend commands run from `apps/backend/` with Maven or Docker Compose
