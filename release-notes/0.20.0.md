# v0.20.0 Release Notes

**Release Date:** June 28, 2026
**Database Schema:** v21 (frontend IndexedDB) + V22 (backend PostgreSQL via Flyway)
**Status:** Minor Release

---

## Overview

v0.20.0 "Production Hardening" closes the two highest-priority security and data-loss
gaps. The refresh token no longer lives in `localStorage` (SEC-002 remediation), and
four entity types that were device-local - recipes, meal templates, reminders, and diet
profiles - now participate in cloud sync.

Web Push notifications (VAPID service-worker dispatcher) were scoped for this release
but deferred; the existing `useReminders` local-timeout approach is unchanged.

---

## Track 1 - SEC-002: HttpOnly Refresh Token

### Problem

The 7-day refresh token was written to `localStorage` as `gc_refresh_token`. Any XSS
payload on the origin could read and exfiltrate it, enabling session hijacking. CWE-312.

### Frontend changes

**`apps/web/src/lib/apiClient.ts`**

- `storeTokens(accessToken, expiresInSeconds)` now takes two arguments. The access token
  is kept in memory only (the `_accessToken` variable). No refresh token is ever stored
  in `localStorage`.
- Session existence is tracked via `sessionStorage.setItem("gc_session", "1")` so
  `isAuthenticated()` returns true across a page reload even before the background refresh
  completes.
- `refreshAccessToken()` calls the new `POST /v1/auth/refresh-cookie` endpoint with
  `credentials: "include"`, letting the browser send the `gc_rt` cookie automatically.
- On load, any legacy `gc_refresh_token` key in `localStorage` is removed automatically.
- All auth fetches (`exchangeToken`, `logout`) use `credentials: "include"`.

### Backend changes

**`apps/backend/.../controller/AuthController.java`**

- `POST /v1/auth/token` now calls `addRefreshCookie(response, auth.refreshToken())` to
  issue a `Set-Cookie: gc_rt` HttpOnly, SameSite=Strict, Secure, 7-day cookie rather
  than returning the refresh token in the JSON body.
- New endpoint `POST /v1/auth/refresh-cookie` reads the cookie via
  `@CookieValue(name = "gc_rt")`, rotates the token, and issues a new cookie.
- `POST /v1/auth/logout` reads the cookie and calls `clearRefreshCookie` to zero out the
  cookie max-age.

**`apps/backend/.../service/AuthService.java`**

- `refresh(String)` performs token rotation: deletes the old JTI before issuing a new
  pair, preventing replay attacks.
- `logout(String)` is null-safe: a blank or absent cookie is a no-op so logout succeeds
  even when the cookie has already expired.

**`apps/backend/.../dto/AccessTokenResponse.java`** (new file)

- Response record returned by `/token` and `/refresh-cookie` carrying only the access
  token and `expiresIn` (seconds). The refresh token is no longer in the JSON body.

## Track 2 - Sync expansion to four additional entity types

### Problem

Recipes, meal templates, reminders, and diet profiles written on one device were never
sent to the cloud. A user switching devices or clearing their browser lost this data
even with cloud sync enabled.

### Schema - Dexie v21

**`apps/web/src/db/dbService.ts`**

- Added `syncId?: string` field to `Recipe`, `MealTemplate`, `Reminder`, and `DietProfile`
  interfaces.
- Extended `SyncEntityType` union with `"recipe"`, `"mealTemplate"`, `"reminder"`,
  `"dietProfile"`.
- `db.version(21)` adds a `syncId` index to the four tables and runs an upgrade callback
  that backfills `crypto.randomUUID()` for any existing rows without a `syncId`, so the
  first sync after upgrade can match them by `syncId`.
- `saveRecipe()` now auto-assigns a `syncId` on new recipes if one is not provided.

### Pull logic

**`apps/web/src/hooks/useSyncService.ts`**

- New exported functions: `pullRecipes`, `pullMealTemplates`, `pullReminders`,
  `pullDietProfile`. Each fetches the entity list from the backend and upserts locally.
- `enqueueAllLocalData` now covers all 10 entity types (was 6). The four new types are
  queued alongside food logs, water logs, activity, body, step, and fasting data when a
  user first activates E2E sync.
- `applyRemoteDelete` and `applyRemoteUpsert` handle the new entity types.

### Write-path enqueue

Every write that mutates the four new entity types now calls `enqueueSyncOperation`:

- **`apps/web/src/state/slices/recipeSlice.ts`** - `deleteRecipe` and `updateRecipe`
  enqueue `"delete"` and `"update"` operations respectively, keyed by `syncId`.
- **`apps/web/src/state/slices/settingsSlice.ts`** - `saveDietProfile`, `saveReminder`,
  and `deleteReminder` enqueue the appropriate operation.
- **`apps/web/src/hooks/useRecipeForm.ts`** - recipe create and edit both enqueue via
  `enqueueSyncOperation`; new recipes receive a `syncId` at form-submit time.
- **`apps/web/src/state/slices/settingsSlice.ts`** - `addMealTemplate`,
  `updateMealTemplate`, and `deleteMealTemplate` enqueue `"create"`, `"update"`,
  and `"delete"` operations respectively.

## Bug Fixes

- `apiClient.ts`: legacy `gc_refresh_token` localStorage key is removed on load,
  preventing stale tokens from lingering after upgrade.
- `AuthService.logout`: null/blank refresh token no longer throws; treated as a no-op
  so cookie-based logout succeeds when the cookie has already expired.

---

## Breaking Changes

- **Auth response shape**: `POST /v1/auth/token` no longer returns `refreshToken` in
  the JSON body. Any client reading `data.refreshToken` from this endpoint must be
  updated to read the `gc_rt` cookie instead.
- **`storeTokens` signature**: changed from three arguments
  `(accessToken, refreshToken, expiresInSeconds)` to two arguments
  `(accessToken, expiresInSeconds)`. The refresh token parameter is removed.
- **Existing sessions**: users already signed in will be signed out on first page load
  after deploying this version. The legacy `gc_refresh_token` localStorage key is
  deleted; without the new `gc_rt` HttpOnly cookie, `isAuthenticated()` returns false
  and the app routes to the sign-in screen. One-time re-authentication is expected.
- **Dexie v21**: the IndexedDB schema upgrades automatically on first load. No data is
  lost; the upgrade callback backfills `syncId` for existing rows.

## Known gaps

- **Web Push / VAPID**: `useReminders` still uses local `setTimeout`. Real push
  notifications that fire when the tab is closed are deferred.
- **Backend sync REST for recipes/templates**: The frontend enqueues operations to the
  existing E2E blob path; plaintext sync REST controllers and Flyway migrations for
  recipes/templates on the backend are deferred.

---

## Upgrade Notes

No Flyway migration required (Flyway stays at V22; no backend schema changes this
release). IndexedDB upgrades automatically on first page load via the Dexie v21
callback.

Existing logged-in users will be signed out once on first load - this is expected
behavior during the SEC-002 migration (see Breaking Changes above).

Pull, run `pnpm install` (no new frontend dependencies), then restart the dev server.
