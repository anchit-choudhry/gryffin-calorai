# v0.19.0 Release Notes

**Release Date:** June 27, 2026
**Database Schema:** v20 (frontend IndexedDB) + V22 (backend PostgreSQL via Flyway)
**Status:** Minor Release

---

## Overview

v0.19.0 "Effortless Capture" ships on-device AI food logging. A user can photograph a
plate, type a free-text meal description, or pipe a voice transcript through the same
parse pipeline and receive an itemized, editable food log with macros grounded in the
Open Food Facts database - all without a network call or API key.

The AI runs locally via transformers.js (Xenova/food-101 classifier) rather than the
cloud backend originally planned. This means: zero server cost, near-zero latency,
full offline support, and no user data leaving the device. The backend `AiRecognitionController`
and `V23__ai_usage.sql` monthly quota are deferred to a future cycle.

No database schema changes this release (IndexedDB stays at v20, Flyway at V22).

## Track 1 - On-Device AI Classifier

### Frontend - Complete (June 2026)

**`apps/web/src/lib/localFoodClassifier.ts`** (new file)

- Wraps `@xenova/transformers` pipeline for `zero-shot-classification` with the
  Xenova/food-101 model; lazy-loads on first call via a module-level singleton
- Singleton reset on pipeline failure: `.catch(() => { pipelinePromise = null; })`
  prevents the poisoned-promise bug where a rejected Promise (truthy) blocks retries
- `classifyFood(imageFile): Promise<RecognizedFoodItem[]>` - returns sorted results
  with `confidence`, `source: "local_ai"`, and calories/macros from the OFF lookup

**`apps/web/src/lib/aiLoggingApi.ts`** (new file)

- `recognizePhoto(file: File): Promise<RecognizedFoodItem[]>` - delegates to
  `classifyFood`; catches per-candidate errors with `continue` so one bad item does
  not abort the whole result set
- `parseText(text: string): Promise<RecognizedFoodItem[]>` - splits free text on
  commas and conjunctions, classifies each token against the local model
- `RecognizedFoodItem` type: `name`, `confidence`, `source`, `offProductId?`,
  `calories?`, `protein?`, `carbs?`, `fat?`

**`apps/web/vite.config.ts`**

- Added `vendor-local-ai` manual chunk covering `@xenova/transformers`; keeps the
  849 KB WASM + model loader out of the main bundle and the existing vendor chunks
- CSP extended: `wasm-unsafe-eval` added to `script-src`;
  `https://huggingface.co` and `https://cdn.jsdelivr.net` added to `connect-src`
  for first-load model weight fetch

**`apps/web/public/_headers`** + **`apps/web/index.html`**

- Same CSP additions applied to Netlify/Cloudflare production headers and the
  meta-tag fallback respectively

## Security Hardening

- **AI consent bypass prevention:** `setAiEnabled(true)` is a no-op without prior
  consent; `loadAiEnabled` enforces the dual-key check (`gc_ai_enabled` +
  `gc_ai_consent`) so a stale localStorage flag cannot re-enable AI after revocation
- **CSP for WASM and CDN origins:** `wasm-unsafe-eval` added to `script-src` (required
  by ONNX Runtime Web); `https://huggingface.co` and `https://cdn.jsdelivr.net` added
  to `connect-src` for first-load model weight fetch
- **Pre-download consent gating:** `AiLoggingPanel` starts the model download only
  after the user taps the explicit consent button - no network call fires on page load
- **Classifier singleton reset on failure:** `localFoodClassifier` resets
  `pipelinePromise = null` in the pipeline catch handler so a failed initialization
  allows the user to retry rather than being permanently blocked

## Upgrade Notes

No database migration required. Pull, run `pnpm install` (new dependency:
`@xenova/transformers`), and restart the dev server.

The CSP `meta` tag in `index.html` and `public/_headers` both include the new
`wasm-unsafe-eval` directive and the two CDN origins. If you serve the app from a
custom CDN or reverse proxy that sets its own `Content-Security-Policy` header, add
`wasm-unsafe-eval` to `script-src` and `https://huggingface.co
https://cdn.jsdelivr.net` to `connect-src` in that header.

On first use of AI logging, the app downloads the Xenova/food-101 model weights
(~20 MB) from `cdn.jsdelivr.net`. Subsequent sessions use the browser cache.
